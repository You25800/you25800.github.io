第一部分：核心理念 ​

1.1 为什么需要结构化提示词 ​

传统方式的问题 ​

❌ 问题场景:

bash
开发者: "帮我写一个订单功能"
AI: (生成 500 行代码)
开发者: "这个不对，我需要支持会员折扣"
AI: (重新生成 500 行代码)
开发者: "还要支持免费次数..."
AI: (再次重新生成...)

问题:

  • 💸 浪费大量 token
  • 🔄 反复返工
  • 😰 代码质量不可控
  • ⏰ 效率低下

结构化方式的优势 ​

✅ 正确场景:

bash
【分析阶段】
开发者: "讨论订单功能需求，不要写代码"
AI: "理解需求，识别 4 种计费场景..."
开发者: "理解正确"

【设计阶段】
开发者: "给出 2-3 个实现方案，不要写代码"
AI: "方案 A: 统一入口... 方案 B: 分散处理..."
开发者: "采用方案 A"

【实施阶段】
开发者: "按方案 A 开始编码"
AI: "生成符合预期的代码"
开发者: "完美！"

优势:

  • ✅ 一次到位，减少返工
  • ✅ 节省 token 成本
  • ✅ 代码质量可控
  • ✅ 效率大幅提升

1.2 三段式工作流 ​

这是 Claude Code 协作的核心方法论：

bash
┌─────────────────────────────────────────┐
│  阶段 1: 分析 (Analysis)                 │
│  目标: 理解现状、评估可行性               │
│  关键词: 解释、分析、不要写代码           │
│  时长: 1-2 轮对话                        │
├─────────────────────────────────────────┤
│  阶段 2: 设计 (Design)                   │
│  目标: 制定方案、确认需求                 │
│  关键词: 讨论、对比、复述、不要写代码      │
│  时长: 2-3 轮对话                        │
├─────────────────────────────────────────┤
│  阶段 3: 实施 (Implementation)           │
│  目标: 编写代码、分步执行                 │
│  关键词: 采用方案 X、开始编码            │
│  时长: 1-N 轮对话（根据复杂度）           │
└─────────────────────────────────────────┘

核心原则 ​

原则 1: 先分析，再设计，后实施

  • ❌ 不要直接要求写代码
  • ✅ 先理解现状，讨论方案，最后编码

原则 2: 关键时刻说"不要写代码"

  • 分析阶段：必须加
  • 设计阶段：必须加
  • 实施阶段：不需要

原则 3: 使用 @ 精确引用

  • ❌ "在订单文件的 100 行左右"
  • ✅ @service/order.go#L100-120

1.3 为什么这样更高效？ ​

效果对比 ​

维度传统方式三段式工作流
返工次数3-5 次0-1 次
Token 消耗大量节省 40-60%
对话轮数15-20 轮5-10 轮

真实案例 ​

传统方式: 12 轮对话，3 次重写 三段式: 5 轮对话，一次到位

节省时间: 70%

第二部分：必备技能 ​

2.1 核心语法 ​

语法 1: @ 文件引用符 ​

作用: 精确引用文件或代码段

格式:

bash
@文件路径                    # 引用整个文件
@文件路径#L50                # 引用第 50 行
@文件路径#L50-80             # 引用第 50-80 行
@文件1 @文件2                # 引用多个文件

示例:

bash
# 引用整个文件
@api/v1/order.go

# 引用特定行
@service/user.go#L100

# 引用范围
@service/order.go#L50-80

# 引用多个文件
@api/v1/order.go @service/order.go

为什么重要？

bash
❌ 错误: "订单文件的第 100 行左右那个函数"
   - AI 可能找错位置
   - 描述模糊不准确

✅ 正确: "@service/order.go#L100-120"
   - 精确定位
   - 不会出错

语法 2: 关键修饰符 ​

修饰符 1: "不要写代码"

bash
用途: 限制 AI 只分析讨论，不生成代码
使用场景: 分析阶段、设计阶段

示例:
@service/order.go#L50-100
解释一下订单创建的完整流程，不要写代码

修饰符 2: "先说明"

bash
用途: 要求 AI 先给出方案，等确认后再执行
使用场景: 复杂修改前、风险较高的操作

示例:
@api/v1/user.go#L50-80
删除 deprecated 字段，先说明影响范围，不要写代码

修饰符 3: "直接修改"

bash
用途: 跳过讨论，立即执行
使用场景: 简单操作、已经明确的修改

示例:
@model/user.go#L20
将 UserID 改为 UID，直接修改

语法 3: / 命令符 ​

bash
/clear          清空上下文
/think          深度思考模式

使用场景:

bash
/clear 使用时机:
  ✅ 切换到完全不同的任务
  ✅ 上下文超过 50 轮对话
  ✅ AI 开始混淆之前的内容
  ❌ 任务进行中（会丢失上下文）
  ❌ 仅仅换了个文件

/think 使用时机:
  ✅ 复杂的架构设计
  ✅ 难以抉择的技术选型
  ✅ 需要深度分析的问题

语法 4: 结构化描述 ​

使用标识符组织信息：

bash
【需求背景】
现有功能管理页面混乱，需要重新设计

【功能目标】
实现统一的功能管理，包括：
- 价格配置
- 免费次数设置
- 平台支持

【技术要求】
- 响应时间 < 500ms
- 支持批量导入
- 配置变更要有审计日志

为什么使用结构化？

  • ✅ 信息清晰，不遗漏
  • ✅ AI 更容易理解
  • ✅ 便于后续追溯

语法 5: 占位符约定 ​

在模板中使用占位符：

bash
{文件}     → api/v1/user.go
{行号}     → #L100-120
{功能}     → 订单创建逻辑
{模块}     → user_benefit
{方案X}    → 方案A
{字段}     → member_level

示例:

bash
模板:
@{文件}#{行号} 解释{功能}，不要写代码

实际使用:
@api/v1/order.go#L50-100 解释订单创建逻辑，不要写代码

5.4 常见问题 ​

Q1: 我应该什么时候使用三段式？ ​

答:

  • ✅ 复杂度 > 简单修改时
  • ✅ 不确定如何实现时
  • ✅ 涉及多个文件协作时
  • ❌ 非常简单的操作（如改个变量名）可以直接要求

Q2: 每次都要三个阶段吗？ ​

答: 不一定

  • 简单任务: 分析 + 实施（2 轮）
  • 中等任务: 分析 + 设计 + 实施（3-5 轮）
  • 复杂任务: 完整三段式（6-10 轮）
